00 · thesis

Files are for things humans author. AI artifacts are something else. v1 doesn't try to replace the filesystem — it sits next to it and turns the ones worth sharing into URLs.

The pain we're killing

  • Tab sprawl — one browser tab per artifact, refreshed by hand.
  • Where-did-I-save-it — Claude names files plan_v3_final.html.
  • Sharing friction — markdown can't be sent, HTML files don't have URLs.
  • Update dance — agent edits the file, you reload, the link you sent goes stale.

What v1 promises

  • No install. A URL you visit in Chrome.
  • One folder, one app tab. Everything else is a link.
  • One click to publish. URL appears, auto-copies, paste in Slack.
  • Republishing keeps the same URL. The link you sent still works after Claude edits.

01 · the user journey

One folder. One app tab. The rest is paste-the-link.

Open the app in Chrome / Edge / Arc / Brave

No install. The app is a single static page. First visit asks for a folder, granted via showDirectoryPicker(). The handle is stored in IndexedDB so the next visit skips the prompt.

Point it at your Claude Code projects folder

The app walks the folder and lists every .html file, newest first. Subfolders are shown collapsed. FileSystemObserver (Chrome 133+) pushes change events — no polling, no refresh button.

Claude generates onboarding_v3.html — it appears in the list instantly

A "new" badge on the row. Click the filename to preview it inline (sandboxed iframe). The preview reads the file directly from disk via the handle — nothing has been uploaded yet.

Click Publish — URL appears and is auto-copied

Toast slides in: "Copied. art.artifaq-user.net/a/k3n9pQr2Lm4x". No modal, no naming step. The URL is the receipt. Paste it in Slack while it's still warm.
https://art.artifaq-user.net/a/k3n9pQr2Lm4x

Teammate opens the link — no account, no app, no install

They see the artifact, full-bleed, with a slim top bar (filename, last-updated, copy-link, view-source). It works on phones. It works in Safari. It works for the PM who never installed your tool.

Claude edits the file — your app shows "changed"

The row gets an orange dot. Click Update — the same URL now serves the new content. The link you pasted in Slack hours ago still works; your teammate just refreshes.

That's the whole product

No canvas. No comments. No projects. No accounts. The folder is the project. The URL is the share button. The toast is the celebration.

02 · architecture

Three pieces. One bill. The user's machine on the left, two domains on the right, R2 in the middle.

Client

Vite + React, ~40KB gz. Talks to disk via File System Access API. Talks to backend via one POST.

Backend

One Worker. POST /publish, GET /raw/:id, GET /a/:id. R2 PUT/GET. ~80 lines.

Serving

A separate registered domain. The app and the artifacts can never read each other's cookies.

03 · key design decisions

Every "yes" here is a "no" to three other options. The why matters more than the what — it's what tells you whether to change the answer when conditions change.

Chromium-only — or build a Safari/Firefox fallback? Chromium-only

Why: File System Access API + FileSystemObserver simply do not exist outside Chromium and Mozilla has said no. The "fallback" — drag-and-drop or <input webkitdirectory> — gives you a one-shot snapshot with no watching, no persistence, no re-prompts. That's not a degraded version of the product, it's a different (worse) product.

Implication: Excludes ~30% of the global market but our target audience (devs using Claude Code) is overwhelmingly on Chrome/Arc/Brave/Edge. Better to be excellent for 70% than mediocre for 100%. Show a friendly browser-check screen with one-click "open in Chrome" intent.

When to revisit: If Apple ships it (don't hold your breath). Or if the audience widens to non-developers.

Auto-publish every file — or let the user pick which ones? User picks

Why: Auto-publish is the AI-tool default and it's wrong. 18 of every 20 generations are noise. Auto-publish means every half-baked exploration leaks to a server. It also forces an account ("whose noise is this?") and burns trust.

Implication: The local app is private by default — nothing leaves your machine until you press a button. This is the biggest single difference between our product and "Claude artifacts in the cloud."

Stable URL on republish — or immutable per-version URL? Stable URL per local file

Why: The article's whole pitch was "I'm increasingly not editing files myself, Claude is." That means the file iterates. If every iteration mints a new URL, every paste in Slack rots within an hour. The article's emphasis on hot-reload tells us live-updating is the value, not version pinning.

How: Each local file path gets one slot (12-char nanoid). Re-publishing the same file overwrites that slot's content but keeps the URL. There's also a "Publish as new" affordance for when you explicitly want a frozen version (sending to a client, attaching to a PR).

Trade-off: Immutability is safer for screenshots and forensic clarity. We give that up in exchange for the link surviving Claude's next edit. Vercel's pattern (immutable deploys + a mutable alias) is the right v2.

Login on day one — or anonymous capability URLs? Anonymous, 30-day TTL

Why: Login crushes the funnel and Loom/Excalidraw/file.io all proved the URL itself can be the credential. The path's random 12 chars ≈ 71 bits — unguessable. Bots are kept out with Cloudflare Turnstile and an IP rate-limit, not an account wall.

Lifecycle: Anonymous artifacts auto-expire after 30 days of no access (R2 lifecycle rule). A "claim with email" button lets a user keep things forever — but we don't ask for it on the first click.

When to revisit: The moment people start asking "where are my old artifacts." That's the signal to add the magic-link account, not a day earlier.

Single big HTML — or bundle local assets (CSS, images, JS)? Single file only in v1

Why: ~95% of Claude-generated HTML is self-contained — inline CSS, inline SVG, sometimes a CDN script. The 5% that links to local ./style.css is solvable but adds: dependency walking, blob URL rewriting or a service-worker resolver, asset upload, asset GC, broken-link UX. None of that is the core promise. Detect external local refs at publish time and show a clear "this file references ./logo.png — that won't be included" warning.

v2: Bundle assets via JSZip + multi-object R2 upload. Or rewrite to inline data-URIs (works for images/css under ~2MB).

Same-origin URL — or separate registered domain for artifacts? Separate eTLD+1, always

Why: Serving user-controlled HTML on your app's domain is "essentially one massive XSS vulnerability" (CodePen's own words). The user's <script> can read your app's cookies, localStorage, IndexedDB. This is why GitHub uses githubusercontent.com and CodePen uses cdpn.io — different registrable domains, so the same-origin policy and SameSite cookie rules build a real wall.

Cost: $12/yr for the second domain. Cheapest security measure you'll ever buy.

Folders vs. projects vs. tags as the unit of organization? The folder is the project

Why: The user already organizes their work in folders. Inventing a separate "project" concept duplicates that organization and forces a sync problem we don't need. In v1 the app shows one folder at a time. You can pin multiple folders in the sidebar later, but each is just a saved handle.

Implication: Switching projects = switching folders. Recent folders are saved in IndexedDB. No database needed.

Live hot-reload to teammates — or manual refresh? Manual refresh in v1

Why: Real-time push needs WebSockets, a persistent connection per viewer, presence, and "did the publisher click update yet" logic. None of that is necessary for the core value (the link stays the same, refresh shows latest). Skipping it saves a week.

v2: Server-sent events on the viewer page → toast "new version available, refresh." Cheap to add later, expensive to build prematurely.

05 · the feel

Premium isn't ornament. It's restraint. Eight rules we don't break.

01

Calm over loud

One accent color. Low default contrast. The app is stagehands, the artifact is the star. Nothing in our UI should ever pull the eye away from what Claude just made.

02

Type does the work

Hierarchy via weight and size, not boxes and dividers everywhere. Borders are reserved for things you can act on (buttons, inputs). Lists are just well-spaced text.

03

Motion confirms, never decorates

Animation only after a user action — a 200ms slide-up for the toast, a 140ms fade for a dot turning green. Nothing pulses, breathes, or animates on load. No skeleton shimmer.

04

Single column, no panels

The file tree is the whole app. Preview opens inline below the row. Settings open inline. No left/right split. No resizable panels. No collapse-toggle for a sidebar that shouldn't exist.

05

Keyboard-first, mouse second

j k to move · space to preview · ↵ to publish · ⌘C to re-copy a live URL · / to filter · ? to see the rest. Discoverable via a quiet "?" badge in the corner — never a tour.

06

State visible, never noisy

A 6px colored dot is the entire status indicator. A faint mono timestamp tells you when. No "MODIFIED" labels, no colored badges, no banners. The signal is always there; you look at it on purpose.

07

The toast is the celebration

Publishing is the one moment that earns motion. The URL slides up from the bottom-right — pre-copied — with a single fading "copied" caption. Three seconds, then gone. No confetti. No modal. No "are you sure."

08

Empty states are full sentences

Not "No files." Instead: "Point this at a folder and we'll list every .html file inside — including nested ones. Drop one here, or pick a folder above." Words do the work boxes can't.

Visual language

Everything in the app is built from this palette and these type sizes. If a screen needs more, we redesign instead of adding.

06 · the file list

The list is the product. Every detail of how it behaves is worth thinking through. Including the small ones.

What you see when you pick a folder

One folder. Walked recursively. Every .html file inside it — including in nested subfolders — laid out as a tree. Folders are expanded by default, with a chevron to collapse. Files are sorted newest-first inside their parent. .git, node_modules, dist, .next are skipped silently — a "show hidden folders" toggle exists for the 2% who need it.

Folder behavior — the small things

  • Default expanded. When you first pick a folder, every subfolder is open. You see everything, immediately. No "click to expand 6 levels."
  • Chevron only on hover for empty folders. Folders with zero .html files inside don't render at all.
  • Collapse state is remembered per-folder. Stored in IndexedDB alongside the handle, so reopening the tab feels like coming back to where you were.
  • Indentation uses dashed guides — not solid lines. Visible enough to follow, quiet enough to ignore.
  • The folder name shows the file count in faint mono next to it, so you know how big it is without expanding.
  • Files at the root appear under a folder labeled (root) only when there are subfolders too — otherwise they just sit at the top.
  • Click anywhere on a folder row to collapse / expand, not just the chevron. The whole row is the affordance.

Status vocabulary — four states only

  • untouched — exists on disk, never published, never previewed. Grey filled dot.
  • new — appeared in the last 60 seconds. Soft orange dot with a faint glow ring. Decays back to untouched after the timer or after you preview.
  • draft — previewed but not yet published. Hollow grey ring.
  • live — published to a URL. Green dot with a soft glow. If the file on disk is newer than what's on R2, the glow grows slightly: "click copy menu → push update."

Search & filter

Sticky input at the top of the list. / focuses it from anywhere. Fuzzy match on filename and folder path. Folders auto-collapse when their entire subtree filters out — no "tree of empties." esc clears the filter and restores prior collapse state.

The publish moment, frame by frame

  1. 0ms — you click publish (or hit ↵).
  2. ~80ms — the request is in flight; the row's dot dims slightly.
  3. ~250ms — response arrives. The dot transitions grey → green over 140ms.
  4. +0ms — toast slides up from bottom-right with the URL, already in your clipboard.
  5. +3s — toast fades. You've already pasted it in Slack. Nothing else moved.

No success modal. No "share to Slack" button. No "view artifact" CTA. The clipboard is the share button.

Empty & edge states

  • No folder yet: a single button — "Pick a folder." That's the entire screen.
  • Folder picked, no HTML inside: "No HTML files in this folder yet. We're watching — anything you (or Claude) drops in will appear here."
  • Permission revoked by OS: soft amber bar at top: "We've lost access to this folder. Click to grant again."
  • Publish failed: toast turns red, contains a retry button. Row stays in draft state.
  • File deleted on disk: row fades to 40% opacity for 5 seconds, then drops out. No "are you sure" — disk is truth.
  • File references local ./assets: a one-time inline note appears under the row at publish time: "this links to ./logo.png, which won't be included." User can publish anyway.

07 · security — the things we get wrong once

Hosting arbitrary user HTML is the riskiest part of this product. Most of the risk evaporates if you make four decisions correctly on day one.

ThreatMitigationWhere
User HTML steals app cookies / localStorage Serve from a different registered domain (different eTLD+1). Same-origin policy + SameSite cookies build the wall. DNS / domain registration
User HTML inherits app session Never set auth cookies on the artifact domain. Ever. Not even for "convenience." Worker
User HTML escapes iframe sandbox in viewer Wrap in <iframe sandbox="allow-scripts allow-forms allow-popups">. Do NOT add allow-same-origin when iframe src is same-origin as parent — it lets the page strip its own sandbox. Viewer page
MIME confusion / served-as-script attacks Content-Type: text/html; charset=utf-8. X-Content-Type-Options: nosniff. Worker response
CSRF / referer abuse Content-Security-Policy: sandbox allow-scripts; on raw artifact responses. Worker response
Bot upload abuse / storage flooding Cloudflare Turnstile on /publish. Workers KV-backed IP rate limit. 30-day TTL. Worker + lifecycle rule
Phishing / brand-impersonation HTML hosted on our URLs Abuse-report endpoint. Manual takedown. Domain is clearly not a primary brand surface. Operational
The rule we never break: the app and the artifacts cannot read each other. Anything you do to "make sharing nicer" that violates this rule is wrong — even if it works for v1.

11 · honest risks

"Why install nothing? Why visit a website? Just give me a CLI." real

Devs sometimes want the opposite of frictionless — they want a tool they own, locally, that pipes to their existing scripts. A CLI version (artifaq publish ./plan.html) is a strong v1.5. The web app and CLI share the Worker backend, so it's additive, not a rebuild.

Chromium-only excludes the long-tail user accepted

Mitigated by the fact that viewing works everywhere — only publishing is gated. Mitigated further by clear messaging on the landing page. Re-evaluate at 1000 weekly active users: if 30%+ are bouncing on the browser gate, ship the drag-drop fallback.

"I already share Claude artifacts via [Claude.ai, GitHub Pages, val.town, Loom, Slack canvas...]" positioning

Each of those has a flaw we exploit: Claude.ai requires Claude.ai. GitHub Pages needs a repo. val.town requires an account & makes URLs containing your username. Loom is video. Slack canvas is in Slack. Our wedge: anything any tool drops in a folder, online in one click. Tool-agnostic + format-agnostic + login-free.

Storage / abuse spikes contained

30-day TTL caps storage growth. Turnstile + per-IP rate limit caps publish floods. 5MB per-artifact cap caps single-blob explosions. R2 free tier holds at 200K active artifacts. If abuse becomes the dominant signal, we ship login on day 60.

The "stable URL on republish" decision breaks for users who want versioning addressed

"Publish as new" exists exactly for them. v2 adds a per-slot history viewer (last 10 versions retained in R2 with date stamps). Until we see that demand, default stays as stable URLs.

v1 is one thing: a website you visit in Chrome that turns selected HTML files in a folder into shareable URLs. Everything we built around that — Worker, R2, sandbox domain, capability URL, FileSystemObserver — exists only because the one thing has to feel effortless.
